### The Crucial Distinction: Balancing vs. Generation

First, let's address the core of your confusion. You need to stop thinking of the `ext_grid` as just another generator. It's fundamentally different.

*   **`p_conv` (Conventional) & `p_ren` (Renewable):** These are **must-take** or **dispatchable generators**. Their active power output (`p_mw`) is physically **always positive or zero**. They only *inject* power.
*   **`p_ext` (External Grid):** This is the **slack bus**. Its job is to **balance** the entire network.
    *   If `Total Load > Total Local Generation`, `p_ext` will be **positive** (it injects power to make up the deficit).
    *   If `Total Load < Total Local Generation` (e.g., high solar output), `p_ext` will be **negative** (it absorbs the surplus power to prevent over-voltage).

This distinction is the key to solving your problems with normalization and loss calculations.

### Answering Your Core Questions

#### 1. "How do you calculate loss which is `generation - load`?"

You've got the right idea. The total active power loss (`P_loss`) in a system is precisely the sum of all injections.

**`P_loss = sum(all P_injections) = sum(P_gen) - sum(P_load)`**

Because of the slack bus, the correct, robust formula for total losses is:

**`Total Loss (MW) = sum(p_ext) + sum(p_conv) + sum(p_ren) - sum(p_load)`**

In your code, the function `_compute_power_balance_violation` calculates the power injections based on voltages and the Ybus matrix (`S_calc_pu`). The *sum of the real part of `S_calc_pu` over all buses* is the total system loss. Your implementation there is correct.

#### 2. "Can total generation be negative?"

Yes, if you define it incorrectly.
*   **Incorrect Definition:** `Total Gen = p_ext + p_conv + p_ren`. This can be negative if `p_ext` is absorbing a large amount of surplus power.
*   **Correct Definition (for physical calculations):** You should define two separate concepts:
    *   **"Local Generation"** or **"Controllable Generation":** `P_local_gen = sum(p_conv) + sum(p_ren)`. This value is **guaranteed to be positive or zero**.
    *   **"Net Injection from Slack":** `p_ext`. This can be positive or negative.

#### 3. "Does it mean a normalized value will be negative?"

A normalized value **should not be negative** if it represents a physical ratio (e.g., loss as a percentage of load). A negative normalized value is a sign that your **denominator** can be negative, which is what you must avoid.

**Never normalize by a value that can be negative or zero.**

*   **Bad Denominator:** `P_local_gen + P_ext`. Can be negative.
*   **Good Denominator:** `P_total_load`. Total system load is always positive in any meaningful scenario. This is the most robust choice for normalization.
*   **Okay Denominator:** `P_local_gen`. It's always positive, but could be zero if a system has no conventional/renewable generators, leading to division-by-zero errors. `P_total_load` is generally safer.

#### 3. The Loss Function and Metrics (`metrics.py`)

Your `PowerSystemLoss` class is well-structured. My previous advice holds and is critical for publication.

**The Problem:** Normalization and physical calculations are good but can be more robust.

**The Right Way & Solutions:**

1.  **`_compute_normalized_active_power_loss`**: Your current implementation is complex and has multiple versions. The most direct and physically sound method is based on the principle of conservation of energy: **Total Loss = Total Injection**.

    ```python
    def _compute_normalized_active_power_loss(self, state: torch.Tensor, Ybus: torch.Tensor, epsilon: float = 1e-9) -> torch.Tensor:
        # state is the denormalized tensor
        vm_pu, va_rad = state[..., 0], state[..., 1]
        V = vm_pu * torch.exp(1j * va_rad)

        # S_calc = V * conj(I) = V * conj(Ybus * V)
        I = torch.einsum('bij,bj->bi', Ybus.cfloat(), V)
        S_injections = V * torch.conj(I)

        # Total active power loss is the sum of all real power injections. This is always positive in a real system.
        total_loss_pu = torch.sum(S_injections.real, dim=-1)

        # NORMALIZE BY TOTAL LOAD. This is the most stable and physically meaningful denominator.
        p_load_mw_total = torch.sum(state[..., 2], dim=-1)
        total_load_pu = p_load_mw_total / self.s_base_mva

        # Avoid division by zero when load is zero (unlikely but possible)
        normalized_loss = total_loss_pu / (total_load_pu + epsilon)
        
        return normalized_loss
    ```

**The Right Way: Enforce a Consistent Data Shape**

The normalizer should not be responsible for guessing shapes. The code that calls it should ensure the data is in the correct, structured format.

**Solution:** Simplify `denormalize` to only work on 3D tensors.

```python
class PowerSystemNormalizer:
    # ... __init__ and normalize are fine ...

    def denormalize(self, data: torch.Tensor) -> torch.Tensor:
        # Expects data to be in the shape [batch_size, num_nodes, num_features]
        if data.dim() != 3:
            raise ValueError(f"denormalize expects a 3D tensor, but got {data.dim()}D.")

        num_output_features = data.shape[-1]
        
        # Slicing the mean/std is still needed if the model predicts fewer features than were used for training
        mean_slice = self.mean[:num_output_features]
        std_slice = self.std[:num_output_features]
        
        mean_tensor = torch.from_numpy(mean_slice).float().to(data.device)
        std_tensor = torch.from_numpy(std_slice).float().to(data.device)

        # Simple, direct denormalization
        return data * std_tensor + mean_tensor
```
Then, in your evaluation and training loops, you must ensure you reshape the model's output to `[batch, nodes, features]` *before* calling `denormalize`. This makes the data flow explicit and easier to debug.
